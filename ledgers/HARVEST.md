This is the session the Issue 4 harvest (above) names as its concurrent disruptor — confirmed from this side too: mid-walk edits to `MapEditor.tsx`/`useGeoJsonDrop.ts` landed on the shared checkout while the Issue 4 deletion sweep was running its own test-gated loop. | lesson | general rule | stored where |
|---|---|---|
| `ToastProvider`'s `<ToastContext.Provider value={{ add, dismiss }}>` passed a fresh object literal every render. `add`/`dismiss` are individually `useCallback`-stable, but `useToast()`'s `success`/`error` wrappers depend on `[ctx]` — so every toast (add or auto-dismiss) produced a new `ctx`, new `success`/`error` identities, and re-fired every effect in the tree that listed a `useToast()` callback in its deps (e.g. MapEditor's autosave-wiring effect would have re-run on every toast). | Memoize a `Context.Provider`'s `value` whenever downstream consumers derive `useCallback`-wrapped functions from it via `useContext` — an unmemoized value object silently invalidates every one of those callbacks on each provider re-render, not just ones that read the raw context value. Not caught by `exhaustive-deps` lint. | fixed in code (commit `180b839`, `ToastProvider.tsx`) |
| The vendored Excalidraw `Dialog` rendered its close button only when `isFullscreen` (phone form factor) — so on desktop an error dialog had zero focusable elements: autofocus focused nothing, Escape did nothing, and the only working dismissal was clicking the narrow backdrop margin outside the dialog card. Found only by actually driving the browser (playwright-cli) through the failure path, not by reading the component. | A modal's dismissal affordance (close button, and anything focus-scoped like Escape) must not be conditioned on form factor — every dialog needs at least one always-rendered focusable control regardless of screen size. | fixed in code (commit `694a95c`, `packages/excalidraw/components/Dialog.tsx`) |
| Two feature-shaped dead API surfaces from Direction 1 (`@atlasdraw/data`'s `parseCSV`/`PhotonGeocoder`) turned real the moment they were wired into `useGeoJsonDrop` — same "verdict before delete" principle as Issue 4's CursorOverlay finding, but from the opposite direction: a capability-reach row can resolve to "wire it up," not just "keep or delete." | When a capability-reach audit (Direction 1's `CAPABILITY.md` pattern) finds an orphaned-but-tested module with no gate or "not wired yet" marker, treat "wire it into the UI" as a live verdict option alongside pursue/park/reject, not just a headroom placeholder. | confirmed pattern; no new rule needed — already implicit in the direction loop's pursue verdict |
| Playwright-cli's file-chooser interception throws `AbortError` on `showSaveFilePicker` specifically (it can drive `showOpenFilePicker`/`<input type=file>` via `upload`, but has no equivalent for the save picker) — exercising the app's own FSA-unavailable fallback path required deleting `window.showSaveFilePicker`/`showOpenFilePicker` before triggering Save/Open, forcing the app's real `hasFSA()` feature-detection down the download/input branch. | To browser-test code gated on File System Access API availability with a tool that can't drive the native save picker, delete the API from `window` before the action rather than trying to handle the picker — this exercises the app's actual feature-detection fallback, which is the code path most users (non-Chromium, or FSA-denied) actually take. | technique only; no persistent artifact — noted here for the next FSA-gated walk |

| During MapEditor Cut 2, `eslint --max-warnings=0` (the repo's pre-commit gate) flagged `currentConvertibleSelection`/`handleConvert` as unused in MapEditor after extraction — they'd been dead outside the moved effect since before this session; a stale "W-B MainMenu.Item" comment claimed a UI surface (a MainMenu "Convert" item) that grep confirmed never existed. | Treat the lint gate as a correctness signal during extraction, not just style — an "unused after move" warning usually means the code was already only reachable through the piece you just moved, and any comment claiming otherwise is drift to fix on the spot, not carry forward. | fixed in code (commit `9cf4598`); pattern is now explicit in `useConvertToDataLayer.ts`'s header comment |

Narrowing the extracted method's param type was an unforced error, not a preservation of the original contract. | When extracting a function/method during a refactor, re-derive parameter types from the actual caller and callee signatures (grep the real declarations) rather than eyeballing a plausible-looking type — a real typecheck run catches the mismatch immediately, but only if you don't first paper over it with a cast. | fixed in code (commit `a676219`, `yjsChannel.ts`) |
