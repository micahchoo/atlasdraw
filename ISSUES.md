## Issue 1 — The two places that actually gate access (workspace isolation, the realtime relay) don't

> **Status: done 2026-07-04** — ledger: `SECURITY.md` (branch
> `security/managed-mode-trust-boundary`, merged to `main`). All 9 rows
> resolved, sweep complete, a second pass added no rows. The 3 HIGH rows
> (workspace-scoped read/write, share-token minting, `/yjs` room auth) plus
> 2 LOW rows: **documented + gated, not ACL-enforced** — maintainer verdict
> was self-host-only posture (`MANAGED_MODE` defaults off, warns loudly at
> boot, gap recorded in `docs/security/managed-mode-trust-boundary.md`);
> building real workspace ACLs was explicitly declined as out-of-scope
> feature work, not a partial fix. The 3 MED rows that harden even a
> single-tenant deployment were actually fixed in code: Socket.IO CORS is
> now env-driven (`CORS_ORIGIN`), the realtime relay moved behind Caddy/TLS
> (host port 4001 no longer published), and storage gained a per-IP
> rate-limit middleware (4 new tests, 114 storage tests green).

**Symptom:** `apps/storage/src/routes/maps.ts` tags `workspace_id` on
`POST /maps` (line 33-41) but `GET /maps/:id` (line 47-58) and
`PUT /maps/:id` (line 60-79) call `client.getMap(id)` / `updateMap(id, body)`
with no workspace argument at all — confirmed by direct read, both handlers
take only the id. Both storage adapters' `getMap` (`sqlite-fs.ts:197`,
`postgres-minio.ts:212`) ignore workspace entirely. In managed mode, any
valid `X-Workspace-ID` can read or overwrite any map by knowing its id.
Meanwhile `apps/realtime/src` has **zero** hits for `auth|token|verify|secret`
(confirmed by grep) and `index.ts:34` sets Socket.IO CORS to `origin: "*"`.
`yjs-server.ts:14-22`'s own comment claims "SaaS deployments rely on
workspace ACL at the path-routing boundary" — but `workspaceId` there is
just a client-supplied unvalidated string (`socket-io-server.ts:141-147`).
Rounding out the same trust-boundary sweep: the relay is published directly
on `4001:4001` in `docker-compose.yml:113-114`, bypassing the Caddy/TLS
front door every other route gets; and `apps/storage` has no per-IP/per-client
rate limiting on `POST /maps` or `/share` (only realtime sockets are
throttled, in `rate-limit.ts`).

**Rungs:** L1 (README/ADR claim "workspace ACL," "opaque relay") ↔ L4
(the code that would enforce it doesn't exist). A textbook rung mismatch:
the story two layers up promises isolation the implementation never wrote.

**Why it's high-leverage:** This is the one place in the audit where the
gap isn't cosmetic — it's a live cross-tenant data exposure in the hosted
mode the project explicitly ships (`MANAGED_MODE=true`, per CHANGELOG).
**Teaches:** security auditing, run on a calendar — attackers don't file bug
reports; the sweep is how you hear about a hole before someone walks
through it.

**Loop:** `sweep-triage-fix-resweep`. Ledger `SECURITY.md`: finding | file |
severity | user trust broken | fix commit | re-audit. Scope: `apps/storage/src`,
`apps/realtime/src`, `infra/`. Already-known findings above seed the first
four rows; the sweep should still walk both directories fresh in case more
exist. Done when every finding at or above **medium** severity reads clean
on re-audit and a fresh full sweep of both directories adds no rows.

**Run it:**
```
Audit code/apps/storage/src and code/apps/realtime/src (plus infra/docker-compose.yml
and infra/Caddyfile) directory by directory for injection, auth bypass, unvalidated
input, committed secrets, and unsafe deserialization; for secrets, scan full git
history too (git log -p). Start from these seeded findings and verify/extend them:
(1) GET/PUT /maps/:id in apps/storage/src/routes/maps.ts never checks workspace_id
even though POST tags it; (2) apps/realtime/src has no auth/token verification
anywhere and Socket.IO CORS is origin: "*"; (3) the realtime service is published
directly on host port 4001 in docker-compose.yml, bypassing the Caddy/TLS front
door every other route gets; (4) apps/storage has no per-IP/per-client rate
limiting on POST /maps or /maps/:id/share. Ledger SECURITY.md: finding | file |
severity | user trust broken | fix commit | re-audit. Compile and rank the
complete list before fixing anything; then fix in severity order, one finding
per commit, re-auditing each. A key git history has ever seen is burned: rotate
it at the provider first — deleting the file is theater. Done when everything
at or above medium severity reads clean and a fresh full sweep adds no rows.
```

## Issue 7 — Failures in the save/settings surface are disposed of by disappearing, not by telling anyone

**Symptom:** `openAtlasDocument`/`saveAtlasDocument` (`MapEditor.tsx:275-323`)
catch every failure and route it to `console.warn("[atlasdraw]
openFromDisk failed", err)` / `"saveToDisk failed"` — no toast, no dialog,
nothing the user sees. `state/persistence.ts:531` does the same for
auto-save failures. In the same area, `SettingsDialog.tsx` reads env vars
that don't exist anywhere else in the schema — `VITE_STORAGE_MODE` (line
194) and `VITE_REALTIME_URL` (lines 219-221) — while the real config
(`app-config.ts`) uses `VITE_STORAGE_BASE_URL` / `VITE_REALTIME_WS_URL`; the
dialog's storage status is a hardcoded `<span>Connected</span>` regardless
of actual connection state, and the dialog has no save controls at all —
it's decorative. Rounding out the pattern: the app runs two parallel
notification systems side by side — raw `window.alert` at 4 call sites
(`useExportPNG.ts:30`, `useGeoJsonDrop.ts:68`, `MapEditor.tsx:938,1096`) next
to a `ToastProvider` used by only 2 consumer files.

**Rungs:** L4 (implementation: error disposal) ↔ L2 (the user-visible
behavior when a save silently fails, or opens Settings and sees a lie).

**Why it's high-leverage:** a user who loses a save has no idea it happened
until the map is gone; a user who reads `SettingsDialog` believes wrong
things about their own deployment. **Teaches:** observability — a swallowed
